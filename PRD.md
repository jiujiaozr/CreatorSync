## 2. 产品背景

很多内容创作者需要把同一份内容发布到公众号、知乎、B站、小红书等平台。问题是，每个平台喜欢的内容格式和表达风格都不一样：

- 公众号适合完整长文、清晰段落和正式标题。
- 知乎更看重问题意识、观点和逻辑论证。
- B站需要视频标题、简介、分区、标签和脚本摘要。
- 小红书更偏口语化表达、封面标题、话题标签和种草风格。

如果创作者每个平台都手动改一遍，会很花时间，也容易漏掉标题、标签、摘要、格式这些细节。内容开发助手要解决的就是这个问题：用户输入一份原始内容，系统自动生成适合不同平台的发布版本，并支持预览、编辑和模拟发布。

## 3. 产品目标

### 3.1 核心目标

帮助内容创作者把一份原始内容快速改造成多个平台可发布的版本，减少重复改写、排版和检查的时间。

### 3.2 展示目标

本项目是面向产品经理岗位的实战作品，所以除了能跑，还要展示清楚这些能力：

- 是否理解目标用户和真实痛点。
- 是否能把复杂流程拆成清楚的产品功能。
- 是否能设计一个完整的 MVP。
- 是否考虑了未来扩展更多平台的方式。
- 是否能做出一条顺畅的演示路径。

## 4. 目标用户

### 4.1 主要用户

- 自媒体创作者：一个人运营多个平台，需要提高发布效率。
- 内容运营同学：为品牌、公司或活动做多平台内容分发。
- 内容团队负责人：希望统一管理内容适配和发布状态。

### 4.2 用户痛点

- 同一篇内容要反复改标题、摘要、标签和正文结构。
- 不同平台风格差异大，人工适配成本高。
- 多平台预览分散，很难一次性对比效果。
- 发布前容易遗漏字段，比如标签、封面标题、简介。
- 新增平台时，规则容易写散，不方便维护。

## 5. 产品范围

### 5.1 MVP 必做范围

- 原始内容输入。
- 目标平台选择。
- 生成不同平台的适配内容。
- 多平台预览对比。
- 生成结果可编辑。
- 一键模拟发布。
- 展示平台扩展架构。

### 5.2 暂不做范围

- 暂不接入真实平台发布 API。
- 暂不做真实账号授权。
- 暂不做多人协作审批。
- 暂不做内容数据分析看板。
- 暂不做真实 AI 计费、额度和账号体系。

这些能力可以放到后续版本，MVP 先保证演示稳定、路径完整。

## 6. 核心流程

用户的主流程如下：

1. 输入原始标题和正文。
2. 选择内容类型、目标受众和目标平台。
3. 点击生成。
4. 系统按平台生成不同版本。
5. 用户查看各平台预览并编辑。
6. 用户点击一键发布。
7. 系统进入模拟发布流程，展示发布状态和结果。

## 7. 功能需求

### 7.1 原始内容输入

优先级：P0

用户可以输入一篇原始内容，作为后续生成的基础。

字段包括：

- 原始标题：必填。
- 原始正文：必填。
- 内容类型：可选，例如知识分享、产品介绍、活动宣传、经验复盘。
- 目标受众：可选，例如职场新人、内容创作者、产品经理、学生。
- 生成偏好：可选，例如更正式、更口语、更有传播感。

验收标准：

- 标题和正文为空时，生成按钮不可用或给出清楚提示。
- 输入内容后，可以进入平台适配流程。

### 7.2 平台选择

优先级：P0

用户可以选择要生成的平台版本。

MVP 支持平台：

- 公众号。
- 知乎。
- B站。
- 小红书。

验收标准：

- 至少可以同时选择 2 个平台。
- 平台选择状态清楚可见。
- 未选择平台时，不能生成或需要提示用户选择。

### 7.3 平台内容生成

优先级：P0

系统根据原始内容和平台规则生成对应版本。

MVP 可以先用 Mock AI，也就是预设规则加模拟生成结果。这样演示更稳定，不依赖 API Key、网络和额度。代码结构要保留真实 AI 接入位置，后续可接入 OpenAI、通义、DeepSeek、豆包等模型。

各平台生成规则：

| 平台 | 生成内容 | 风格要求 |
| --- | --- | --- |
| 公众号 | 标题、摘要、正文排版、结尾引导 | 正式、清晰、适合长文阅读 |
| 知乎 | 问题式标题、观点开头、分段论证、总结 | 逻辑强、观点明确 |
| B站 | 视频标题、简介、分区、标签、脚本摘要 | 适合视频内容和搜索 |
| 小红书 | 封面标题、口语化正文、话题标签、种草式结尾 | 轻松、有分享感 |

验收标准：

- 同一份原始内容能生成不同平台版本。
- 不同平台不能只是换平台名，要体现结构和风格差异。
- 生成过程要有加载状态。
- 生成失败时要有提示，并允许重新生成。

### 7.4 多平台预览

优先级：P0

用户可以在同一页面查看不同平台的预览效果。

预览要求：

- 公众号预览突出标题、摘要、正文段落。
- 知乎预览突出问题标题、观点、分段论证。
- B站预览突出视频标题、简介、标签。
- 小红书预览突出封面标题、正文和话题标签。

验收标准：

- 每个平台的预览样式有明显区别。
- 切换平台时内容不丢失。
- 长内容不会挤坏页面。

### 7.5 内容编辑

优先级：P0

用户可以修改每个平台生成后的内容。

编辑内容包括：

- 标题。
- 摘要或简介。
- 正文。
- 标签。
- 平台专属字段，例如 B站分区、小红书封面标题。

验收标准：

- 用户编辑后，预览内容同步更新。
- 不同平台的编辑内容互不覆盖。
- 刷新前不要求保存到后端，MVP 可以先保存在前端状态里。

### 7.6 一键模拟发布

优先级：P0

用户点击发布后，系统模拟多个平台的发布过程。

发布状态包括：

- 待发布。
- 发布中。
- 发布成功。
- 发布失败。

验收标准：

- 发布前展示确认信息。
- 发布中展示进度或状态变化。
- 发布完成后展示每个平台的结果。
- 至少支持一个平台失败的模拟状态，体现异常处理思路。

### 7.7 平台扩展架构说明

优先级：P0

项目需要说明以后如何扩展到抖音、视频号、微博等平台。

建议使用平台适配器设计。简单说，就是每个平台有自己的一套规则，新增平台时只新增对应配置和处理逻辑，不影响已有平台。

核心结构示例：

```ts
type PlatformAdapter = {
  id: string;
  name: string;
  fields: PlatformField[];
  generate: (input: SourceContent) => PlatformDraft;
  validate: (draft: PlatformDraft) => ValidationResult;
  publish?: (draft: PlatformDraft) => PublishResult;
};
```

验收标准：

- README 或 PRD 中能解释平台适配器的作用。
- 代码中不同平台规则尽量独立。
- 新增平台时，不需要大改主流程。

## 8. 页面需求

### 8.1 工作台首页

首页直接进入产品工作台，不做单纯宣传页。

页面区域：

- 原始内容输入区。
- 平台选择区。
- 生成按钮和状态提示。
- 多平台预览区。
- 编辑区。
- 发布结果区。

### 8.2 推荐布局

桌面端建议：

- 左侧：输入原始内容和选择平台。
- 中间：生成结果列表或平台标签页。
- 右侧：当前平台预览和编辑。

移动端建议：

- 从上到下排列。
- 平台预览用标签页切换。
- 发布按钮固定在合适位置，避免误触。

## 9. 非功能需求

### 9.1 易用性

- 用户第一次打开就能知道从哪里输入内容。
- 按钮文案要直接，比如“生成平台内容”“模拟发布”。
- 错误提示要说人话，例如“请先选择至少一个平台”。

### 9.2 稳定性

- Mock AI 生成不依赖外部服务。
- 即使生成失败，也不能让页面崩掉。
- 发布失败要给出可理解的失败原因。

### 9.3 可扩展性

- 平台规则独立管理。
- 生成逻辑和页面展示尽量分开。
- 真实 AI 和 Mock AI 可以共用同一套输入输出结构。

## 10. 成功指标

MVP 阶段建议用这些指标判断作品是否完成：

- 用户能在 3 分钟内完成一次完整演示。
- 一份内容至少生成 4 个平台版本。
- 用户可以清楚看出 4 个平台版本的差异。
- 发布流程有开始、进行中、成功或失败结果。
- 文档能讲清楚为什么这样设计，以及未来怎么扩展。

## 11. 技术建议

推荐技术方案：

- 前端：React + Vite + TypeScript。
- 样式：Tailwind CSS。
- AI：先做 Mock AI，预留真实 API 接口。
- 数据：MVP 使用前端状态管理即可。
- 发布：MVP 使用模拟发布，不接真实平台。

推荐原因：

- React + Vite 启动快，适合三天内完成。
- TypeScript 能让平台适配器结构更清楚。
- Mock AI 演示稳定，不会因为 API Key 或网络问题翻车。
- 预留真实 API 能体现产品未来落地空间。

## 16. 后续迭代规划

后续迭代采用“先上线，再增强”的顺序。也就是先让作品能被别人通过网址访问，再逐步补上数据库、后端服务和真实平台能力。这样可以先满足作品展示需求，同时避免一次性把范围做得太大。

### 16.1 第二次迭代：真实能力预研版

第二次迭代重点验证真实能力的可行性，但不要求一次性完整上线：

- 预研真实 AI API 的接入方式，明确输入、输出和错误处理。
- 梳理平台 OAuth 授权和真实发布 API 的限制。
- 补充发布失败重试、发布记录、发布前字段校验等产品设计。
- 明确哪些能力必须依赖后端服务，哪些能力仍然可以留在前端。

### 16.2 第三次迭代：免费上线展示版

第三次迭代优先把当前 Demo 免费部署到公网，让任何拿到网址的人都可以访问。

目标：

- 使用 GitHub Pages 免费部署。
- 生成一个公开访问链接，方便放到简历、作品集或提交材料里。
- 保持当前纯前端结构，继续使用 Mock AI 和模拟发布。
- 确保桌面端和移动端都能完成主要演示流程。

本次不做：

- 不新增数据库。
- 不新增后端服务。
- 不新增登录注册。
- 不接入真实 AI API。
- 不接入真实平台发布接口。

验收标准：

- `npm run build` 可以成功执行。
- GitHub Pages 页面可以正常打开。
- 打开公开网址后，可以完成“输入内容 -> 选择平台 -> 生成 -> 编辑 -> 预览 -> 模拟发布 -> 查看结果”的完整流程。
- README 中写清楚线上访问地址、部署方式和部署前检查项。

风险和应对：

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| GitHub Pages 部署后资源路径错误 | 页面可能空白或样式丢失 | Vite 项目根据仓库名配置 `base` |
| 免费托管有缓存延迟 | 新版本可能不是立刻生效 | README 中说明等待几分钟后刷新 |
| 纯前端无法保存用户数据 | 刷新后内容会丢失 | 在第三次迭代中明确这是展示版限制，第四次迭代再解决 |

### 16.3 第四次迭代：后端服务增强版

第四次迭代再引入数据库和后端服务。这里的后端服务，简单说就是运行在服务器上的程序，负责处理前端不适合直接做的事情；数据库就是用来长期保存数据的地方。

后端和数据库能带来的好处：

- 保存内容历史：用户生成过、编辑过的内容不会因为刷新页面丢失。
- 保存发布记录：能看到每个平台发布成功、失败、重试的过程。
- 保护 AI Key：真实 AI 的密钥不能放在前端，后端可以统一保管和调用。
- 支持用户账号：以后可以区分不同创作者或团队成员。
- 支持团队协作：为编辑、审核、发布等流程打基础。
- 支持真实平台授权：OAuth 授权和真实发布接口通常需要后端处理回调和令牌。

建议范围：

- 新增后端服务，提供内容生成记录、发布记录等接口。
- 新增数据库，保存用户内容、平台草稿和发布状态。
- 前端从“只保存在页面状态里”升级为“可以从服务端读取和保存”。
- 真实 AI 和真实平台发布接口继续分阶段接入，不在第四次一口气全部完成。

验收标准：

- 用户可以保存一条生成记录，并在刷新页面后重新打开。
- 每个平台的发布状态可以被记录和查看。
- AI API Key 不出现在前端代码中。
- README 中补充后端启动方式、环境变量和数据库说明。

### 16.4 第五次迭代：DeepSeek 真实 AI 生成版

第五次迭代引入真实 AI，暂定使用 DeepSeek 模型。真实 AI Key 不能放在前端代码里，需要通过后端代理接口调用。

目标：

- 新增一个轻量后端接口，用来保管 DeepSeek API Key 并调用模型。
- 前端生成内容时，可以在 Mock AI 和真实 AI 之间切换。
- 真实 AI 输出继续复用现有的平台草稿结构，保证编辑、预览、发布记录流程不被打断。
- 失败时保留 Mock AI 兜底，避免演示时完全不可用。

推荐实现：

- 前端继续部署在 GitHub Pages。
- 后端可优先使用 Vercel Serverless、Supabase Edge Functions 或 Cloudflare Workers 的免费额度。
- DeepSeek API Key 放在后端环境变量中，不进入前端仓库。

本次不做：

- 不直接把 API Key 写进浏览器代码。
- 不接真实平台发布 API。
- 不做复杂 AI 计费和额度管理。

验收标准：

- 用户可以选择真实 AI 生成平台文案。
- DeepSeek 调用失败时，页面给出清楚提示，并允许回退到 Mock AI。

### 16.5 第六次迭代：真实平台接入预研与半自动发布

第六次迭代不急着一次性接入所有真实平台 API，而是先把真实平台接入的限制、授权状态和发布前准备做清楚。

目标：

- 提供半自动发布能力，例如复制发布内容、打开目标平台发布入口、生成发布清单。

### 16.6 第七次迭代：单平台真实 API 小闭环

第七次迭代选择一个最适合优先接入的平台，先完成单平台真实 API 小闭环。这样可以先验证授权、字段校验、发布失败处理和记录保存是否跑得通。

目标：

- 选择一个平台作为真实接入试点。
- 完成 OAuth 授权、项目级绑定或平台要求的接入方式。
- 支持真实草稿同步或真实发布中的一个小闭环。
- 将真实发布结果写入 `publish_records`。
- 在个人中心提供平台账号接入入口，为后续用户级绑定预留 UI 和类型结构。

平台选择原则：

- 优先选择文档清楚、审核成本低、接口限制少的平台。
- 如果正式发布权限难申请，可以先做“真实草稿同步”或“发布前内容导出”。

验收标准：

- 至少一个平台可以完成真实授权流程或真实草稿同步流程。
- 成功、失败和重试状态可以被记录。
- 不影响其他平台继续使用 Mock/半自动发布。
- 用户能在个人中心看懂哪些平台已试点接入，哪些平台仍待接入。

### 16.7 第八次迭代：多平台真实发布扩展

第八次迭代再考虑把真实接入能力扩展到更多平台。此时系统已经有数据保存、真实 AI、半自动发布和单平台 API 经验，扩展风险会更低。

目标：

- 将真实接入能力扩展到第二个平台或更多平台。
- 抽象统一的授权状态、发布任务、失败重试和记录查询结构。
- 为不同平台保留独立 adapter，避免新增平台时大改主流程。

验收标准：

- 新增平台接入时，不需要重写主工作台流程。
- 多平台真实/半自动发布状态能在同一页面查看。
- README 中清楚说明哪些平台已真实接入，哪些仍是模拟或半自动能力。
